- **promptNode B.3 widening — `initial: []` on messages node REJECTED (locked 2026-04-30 post-Agent-A-review).** The 2026-04-30 task description proposed adding `initial: []` to the messages node "for parity with `streamingPromptNode`." Re-evaluation rejected this:
  - The current SENTINEL-only design is semantically correct per the v5 guard pattern (`feedback_guard_patterns.md`): `undefined` = global SENTINEL, `T | null` = valid DATA, `=== undefined` is a valid sentinel guard. The two-tier semantics are load-bearing — **initial no-input** (no dep DATA'd → first-run gate blocks → outer cache stays `undefined`) is genuinely distinct from **mid-flow no-input** (deps DATA'd then went null → fn emits empty msgs → switchMap dispatches `node([], { initial: null })` → outer emits `null`).
  - Adding `initial: []` primes messagesNode at construction → switchMap fires immediately → empty-msgs branch dispatches its `initial: null` → outer emits phantom `null` BEFORE any LLM call. Re-breaks `harness.test.ts` `queueDepths["auto-fix"]` and `fast-retry exhaustion` (the same regression class that triggered the 2026-04-24 revert at `archive/docs/SESSION-ai-harness-module-review.md:3654`). `withLatestFrom`-paired triggers fire on the phantom null per implementation-plan L4.
  - Streaming/prompt parity is the wrong frame: streaming has no harness queue/retry feedback, so its constraints differ. Forcing parity here breaks load-bearing distinctions.
  - **Decision:** keep messagesNode without `initial:`; do NOT add a parallel "primed" state node either. The describe-parity argument doesn't hold — messagesNode is already named + tagged + visible in `describe()`; only its cache is `undefined` until first dep DATA, which is the correct behavior for first-run-gated nodes throughout the library.
  - **Action item:** update `archive/docs/SESSION-ai-harness-module-review.md` and the implementation plan to remove the `initial: []` ask for messagesNode. Spec-repo `~/src/graphrefly/COMPOSITION-GUIDE-PROTOCOL.md:516, :528` still references `prompt_node::call` — needs alignment to `::response` per the C+D widening rename.

- **Three-layer view model + changeset stream (2026-04-27, design-review).** Architectural decision: the live topology view decomposes into three layers — **(1) Topology** (structure/edges/types via `describe()` → GraphSpec JSON, fully restorable), **(2) Data** (runtime values/causal flow via `observe()` changeset stream, best-effort replay — sync/async/stream timing is inherently lossy), **(3) Functions** (not serializable by design — the library abstracts fn bodies behind causal chains). Layer 1 + Layer 2 together provide the complete live view; Layer 3 is intentionally hidden. Key sub-decisions:
  - **Renderer extraction (D1 — LANDED 2026-04-30).** `describe()` outputs data only (GraphDescribeOutput / GraphSpec). Rendering lives in `@graphrefly/graphrefly/extra/render` as pure functions: `graphSpecToMermaid(desc, opts?)`, `graphSpecToAscii(desc, opts?)`, `graphSpecToD2(desc, opts?)`, `graphSpecToPretty(desc, opts?)`, `graphSpecToJson(desc, opts?)`, `graphSpecToMermaidUrl(desc, opts?)`. The original Tier 2.1 A2 extraction (2026-04-27) shipped these under `to*` names with the `format` sugar still on `describe()`; the D1 follow-up (2026-04-30) **dropped the `format` sugar entirely** (pre-1.0 BREAKING — no shim) and renamed the public functions to the explicit `graphSpecTo*` form. Internal callers (`validate-observability`, `demo-shell`), the in-tree examples (`inbox-reducer`, `harness-refine-hello`, `nestjs/order-flow`), the website docs (`recipes/nestjs-integration.md`, `blog/42-one-describe.md` postscript, `blog/45-ai-first-engineering.md`), and the test suite all migrated. `src/__tests__/graph/graph.test.ts` has a byte-identical fixture regression ("D1 byte-identical fixture pins each graphSpecTo* renderer's output") that locks Mermaid / D2 / Pretty / JSON / MermaidUrl output for a known a→b + `c::leaf` mount fixture so any future renderer drift surfaces as a snapshot failure. `graphSpecToSvg` and `graphSpecToReact` are not yet shipped — they were called out in the original D1 sketch but defer to D3 (`topologyView` / `LayoutFrame`).
  - **Changeset stream on `observe()`.** New structured all-nodes mode emits discrete `GraphChange` events instead of accumulated state. Core types: `data` (with `fromPath`/`fromDepIndex` for edge attribution), `dirty`, `resolved`, `error`, `complete`, `teardown`. Topology types: `node-added`, `node-removed`, `mount`, `unmount`. Batch boundary types: `batch-start`, `batch-end` (groups changes into animation frames). Future extensions: `pause`/`resume`, `invalidate`, `trace`, `resubscribe`, `snapshot`. The changeset stream unifies topology + data inputs for `topologyView` into a single subscription.
  - **Reactive animation (no timers).** Animation is driven by the changeset stream itself — each batch of changes IS the animation frame. Renderer highlights edges/nodes based on `data` changes in the current batch; next batch replaces previous highlights. CSS `transition` handles visual fade — no `setTimeout`, no `fromTimer`, no spec §5.10 violation.
  - **`topologyView(graph)` factory.** Thin graph (patterns/topology-view/): `derived("layout")` recomputes Sugiyama on topology changes only; `derived("frame")` merges layout + changeset data events into `LayoutFrame { boxes, edges, changes }`. - **`refineLoop` persistent re-seed / reset surface (long-term follow-up to `refineExecutor`, opened 2026-04-23):**
  `refineExecutor` ships in the harness EXECUTE slot with per-item `switchMap` lifecycle (Option A — per-item mount-and-cancel-on-supersede). This is clean but sacrifices cross-item learning: the strategy model (e.g. `errorCritique`'s accumulated failure samples) resets on every triaged item; convergence priors can't carry over from related items with the same `rootCause`. Option B (persistent loop + re-seed) is the long-term best shape.

  - `refineLoopGraph.setSeed(next: T)` — swap seed, reset `iteration` to 0, append a session-boundary marker to `history`, keep `strategy` scaffolding intact. ## Implementation anti-patterns

Cross-cutting rules for reactive/async integration (especially `patterns.ai`, LLM adapters, and tool handlers). **Keep this table identical in both repos' `docs/optimizations.md`.**

| Anti-pattern | Do this instead | Spec ref |
|--------------|-----------------|----------|
| **Polling** | Do not busy-loop on `node.get()` or use ad-hoc timers to poll for completion. Wait for protocol delivery: `subscribe` / `firstValueFrom` / `first_value_from` patterns on the node produced by `fromAny` / `from_any`. | §5.8 |
| **Imperative triggers** | Do not use event emitters, callbacks, or `setTimeout` + manual `set()` to trigger graph behavior. All coordination uses reactive `NodeInput` signals and message flow through topology. If you need a trigger, create a reactive source node. | §5.9 |
| **Raw Promises / microtasks (TS)** | Do not use bare `Promise`, `queueMicrotask`, `setTimeout`, or `process.nextTick` to schedule reactive work. Async boundaries belong in sources (`fromPromise`, `fromAsyncIter`) and the runner layer. | §5.10 |

**Audit context:** Wave 4 of the `src/extra/` hardening pass covers `reactive-map`, `reactive-list`, `reactive-index`, `reactive-log`, `pubsub`. Stress tests shipped in `src/__tests__/extra/reactive-map-stress.test.ts` and `reactive-list-stress.test.ts`. **Refactor plan — three-layer approach:**

1. **This round (Wave 4 shipment):** Apply version-counter + lazy-snapshot across all five structures. The version counter becomes the canonical equality signal; identity equality (`a === b` on always-new snapshots) retires. Ships the `size`-emits fix and `setMany`/`appendMany` bulk APIs at the same time, since all depend on the same shared plumbing.
2. **Post-1.0 (follow-up):** Op-log changesets (option A). Emit `{ version, ops: [...], rootRef? }` instead of full snapshots. Composes with version counter from step 1 (version field = the counter). Protocol-level shift requiring consumer reconciliation logic — worth doing once the primitive layer is stable.
3. **Always optional (user-pluggable):** Persistent data structures (option C — RRB-tree for lists, HAMT for maps) via user-supplied backend. Library bundles only the native backend (~30-50 LOC per structure). Users who need structural sharing / time travel / cheap snapshot plug in `@funkia/list`, `immutable-js`, `mori`, or their own via the `*Backend` interface. We may ship reference adapters as separate packages (`@graphrefly/backends-rrb`, `@graphrefly/backends-hamt`) post-1.0 without adding to core bundle size.

**Rationale:** Bundled RRB-tree / HAMT would add ~5-15 KB gzipped to core for a feature not every user needs. The `*Backend` interface costs ~30-50 LOC and zero runtime overhead (native backend is a thin pass-through). Users opting into persistent semantics pay the bundle cost; everyone else gets the same small core they have today.

**Cross-cutting semantic finding — subscriber callback delivery contract:**
Verified by stress test `S7`. The `downWithBatch` delivery rules (`batch.ts:245-310`) — not previously surfaced in user-facing docs — are:

1. **Within one `down()` call, same tier** → messages coalesced into one array, delivered in one `sink()` call. E.g., `down([[DIRTY], [DIRTY]])` produces one `sink([[DIRTY], [DIRTY]])`.
2. **Within one `down()` call, different tiers** → one `sink()` per non-empty tier group (immediate, deferred, terminal). E.g., `down([[DIRTY], [DATA, 42]])` produces `sink([[DIRTY]])` then `sink([[DATA, 42]])`.
3. **Across different `down()` calls (same batch)** → always separate `sink()` invocations, never merged. Each queued closure in `pendingPhase2` is its own sink call.
4. **No semantic dedup** of repeated idempotent messages (e.g., 5 DIRTYs stay 5 DIRTYs).

Implication: `pushSnapshot()` today uses two separate `down()` calls (one for DIRTY, one for DATA), producing 2 sink invocations per mutation. Even rewriting to a single `down([[DIRTY], [DATA, v]])` produces 2 sink invocations (tier split). Batching helps with ordering/glitch-freedom and DATA drain timing — it does NOT reduce callback count. Bulk APIs (`appendMany`, potential `setMany`) are the only way to collapse N mutations into 1 DIRTY + 1 DATA pair (2 callbacks total instead of 2N). This contract should be documented in the `subscribe` / `node.down` JSDoc.
