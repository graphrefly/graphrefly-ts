- **`refineLoop` persistent re-seed / reset surface (long-term follow-up to `refineExecutor`, opened 2026-04-23):**
  `refineExecutor` ships in the harness EXECUTE slot with per-item `switchMap` lifecycle (Option A ‚Äî per-item mount-and-cancel-on-supersede). This is clean but sacrifices cross-item learning: the strategy model (e.g. `errorCritique`'s accumulated failure samples) resets on every triaged item; convergence priors can't carry over from related items with the same `rootCause`. Option B (persistent loop + re-seed) is the long-term best shape.
  **Long-term fix:** add a persistent-session surface to `refineLoop`:
  - `refineLoopGraph.setSeed(next: T)` ‚Äî swap seed, reset `iteration` to 0, append a session-boundary marker to `history`, keep `strategy` scaffolding intact.   - `refineLoopGraph.reset(mode?: "soft" | "hard")` ‚Äî soft = keep strategy object + any effectivenessTracker totals; hard = full state wipe.

  **Design work required:** (a) history-per-item vs history-across-items semantics ‚Äî probably `historyByEpoch: Node<readonly EpochHistory<T>[]>` where one epoch = one seed; (b) `best` semantics across re-seed ‚Äî reset to null, or persist as all-time best with an `epochBest: Node<T | null>` companion; (c) `attachStorage` checkpoint shape must encode epoch boundaries so restore is deterministic; (d) interaction with `pause()` ‚Äî does re-seed on a paused loop queue until resume? (e) `setStrategy` already exists and is orthogonal ‚Äî confirm they compose.
  **Blocked on:** `refineExecutor` (Option A) shipping first so we have a concrete consumer driving the re-seed API. Schedule as a follow-up session once the harness dogfood example surfaces real cross-item learning opportunities worth the design surface cost.

## Implementation anti-patterns

Cross-cutting rules for reactive/async integration (especially `patterns.ai`, LLM adapters, and tool handlers). **Keep this table identical in both repos' `docs/optimizations.md`.**

| Anti-pattern | Do this instead | Spec ref |
|--------------|-----------------|----------|
| **Polling** | Do not busy-loop on `node.get()` or use ad-hoc timers to poll for completion. Wait for protocol delivery: `subscribe` / `firstValueFrom` / `first_value_from` patterns on the node produced by `fromAny` / `from_any`. | ¬ß5.8 |
| **Imperative triggers** | Do not use event emitters, callbacks, or `setTimeout` + manual `set()` to trigger graph behavior. All coordination uses reactive `NodeInput` signals and message flow through topology. If you need a trigger, create a reactive source node. | ¬ß5.9 |
| **Raw Promises / microtasks (TS)** | Do not use bare `Promise`, `queueMicrotask`, `setTimeout`, or `process.nextTick` to schedule reactive work. Async boundaries belong in sources (`fromPromise`, `fromAsyncIter`) and the runner layer. | ¬ß5.10 |
| **Raw async primitives (PY)** | Do not use bare `asyncio.ensure_future`, `asyncio.create_task`, `threading.Timer`, or raw coroutines for reactive work. Async boundaries belong in sources (`from_awaitable`, `from_async_iter`) and the runner layer. | ¬ß5.10 |
| **Non-central time** | Do not schedule periodic work with raw `setTimeout` / `setInterval` / `time.sleep` for graph-aligned sampling. Use `fromTimer` / `from_timer` (or other documented `extra` time sources) and compose reactively. Use `monotonicNs()` / `monotonic_ns()` for event ordering, `wallClockNs()` / `wall_clock_ns()` for attribution. | ¬ß5.11 |
| **Hardcoded message type checks** | Do not hardcode `if (type === DATA)` for checkpoint or batch gating. Use `messageTier` / `message_tier` utilities for tier classification. | ¬ß5.11 |
| **Bypassing `fromAny` / `from_any` for async** | Do not one-off `asyncio.run`, bare `.then` chains, or manual thread sleeps to bridge coroutines / async iterables / Promises into the graph. Route unknown shapes through `fromAny` / `from_any` so `DATA` / `ERROR` / `COMPLETE` stay consistent end-to-end. | ¬ß5.10 |
| **Leaking protocol internals in Phase 4+ APIs** | Domain-layer APIs (orchestration, messaging, memory, AI, CQRS) must never expose `DIRTY`, `RESOLVED`, DepRecord, or settlement internals in their primary surface. Use domain language. Protocol access available via `.node()` or `inner`. | ¬ß5.12 |
| **`Node` resolution without `get()`** | When blocking until first `DATA`, prefer `node.get()` when it already holds a settled value, then subscribe only if still pending ‚Äî avoids hangs when the node does not replay `DATA` to new subscribers. | ‚Äî |
| **Passing plain strings through `fromAny` (TypeScript)** | `fromAny` treats strings as iterables (one `DATA` per character). For tool handlers that return plain strings, return the string directly; use `fromAny` only for `Node` / `AsyncIterable` / Promise-like after await. | ‚Äî |

### Reactive data structures (Wave 4 audit, 2026-04-15)

**Audit context:** Wave 4 of the `src/extra/` hardening pass covers `reactive-map`, `reactive-list`, `reactive-index`, `reactive-log`, `pubsub`. Stress tests shipped in `src/__tests__/extra/reactive-map-stress.test.ts` and `reactive-list-stress.test.ts`. The items below are queued as one refactor batch so fix + optimization work across all five data structures ships together.

**Refactor plan ‚Äî three-layer approach:**

1. **This round (Wave 4 shipment):** Apply version-counter + lazy-snapshot across all five structures. Add `*Backend` interfaces (`MapBackend<K,V>`, `ListBackend<T>`, `IndexBackend<K,V>`, `LogBackend<T>`) that default-delegate to native `Map` / `Array`. Reactive emission layer reads `version` from the backend and defers snapshot materialization. The version counter becomes the canonical equality signal; identity equality (`a === b` on always-new snapshots) retires. Ships the `size`-emits fix and `setMany`/`appendMany` bulk APIs at the same time, since all depend on the same shared plumbing.
2. **Post-1.0 (follow-up):** Op-log changesets (option A). Emit `{ version, ops: [...], rootRef? }` instead of full snapshots. Composes with version counter from step 1 (version field = the counter). Protocol-level shift requiring consumer reconciliation logic ‚Äî worth doing once the primitive layer is stable.
3. **Always optional (user-pluggable):** Persistent data structures (option C ‚Äî RRB-tree for lists, HAMT for maps) via user-supplied backend. Library bundles only the native backend (~30-50 LOC per structure). Users who need structural sharing / time travel / cheap snapshot plug in `@funkia/list`, `immutable-js`, `mori`, or their own via the `*Backend` interface. We may ship reference adapters as separate packages (`@graphrefly/backends-rrb`, `@graphrefly/backends-hamt`) post-1.0 without adding to core bundle size.

**Rationale:** Bundled RRB-tree / HAMT would add ~5-15 KB gzipped to core for a feature not every user needs. The `*Backend` interface costs ~30-50 LOC and zero runtime overhead (native backend is a thin pass-through). Users opting into persistent semantics pay the bundle cost; everyone else gets the same small core they have today.

**Cross-cutting semantic finding ‚Äî subscriber callback delivery contract:**
Verified by stress test `S7`. The `downWithBatch` delivery rules (`batch.ts:245-310`) ‚Äî not previously surfaced in user-facing docs ‚Äî are:

1. **Within one `down()` call, same tier** ‚Üí messages coalesced into one array, delivered in one `sink()` call. E.g., `down([[DIRTY], [DIRTY]])` produces one `sink([[DIRTY], [DIRTY]])`.
2. **Within one `down()` call, different tiers** ‚Üí one `sink()` per non-empty tier group (immediate, deferred, terminal). E.g., `down([[DIRTY], [DATA, 42]])` produces `sink([[DIRTY]])` then `sink([[DATA, 42]])`.
3. **Across different `down()` calls (same batch)** ‚Üí always separate `sink()` invocations, never merged. Each queued closure in `pendingPhase2` is its own sink call.
4. **No semantic dedup** of repeated idempotent messages (e.g., 5 DIRTYs stay 5 DIRTYs).

Implication: `pushSnapshot()` today uses two separate `down()` calls (one for DIRTY, one for DATA), producing 2 sink invocations per mutation. Even rewriting to a single `down([[DIRTY], [DATA, v]])` produces 2 sink invocations (tier split). Batching helps with ordering/glitch-freedom and DATA drain timing ‚Äî it does NOT reduce callback count. Bulk APIs (`appendMany`, potential `setMany`) are the only way to collapse N mutations into 1 DIRTY + 1 DATA pair (2 callbacks total instead of 2N). This contract should be documented in the `subscribe` / `node.down` JSDoc.
