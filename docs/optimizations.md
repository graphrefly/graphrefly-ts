## Implementation anti-patterns

Cross-cutting rules for reactive/async integration (especially `patterns.ai`, LLM adapters, and tool handlers). **Keep this table identical in both repos' `docs/optimizations.md`.**

| Anti-pattern | Do this instead | Spec ref |
|--------------|-----------------|----------|
| **Polling** | Do not busy-loop on `node.get()` or use ad-hoc timers to poll for completion. Wait for protocol delivery: `subscribe` / `firstValueFrom` / `first_value_from` patterns on the node produced by `fromAny` / `from_any`. | §5.8 |
| **Imperative triggers** | Do not use event emitters, callbacks, or `setTimeout` + manual `set()` to trigger graph behavior. All coordination uses reactive `NodeInput` signals and message flow through topology. If you need a trigger, create a reactive source node. | §5.9 |
| **Raw Promises / microtasks (TS)** | Do not use bare `Promise`, `queueMicrotask`, `setTimeout`, or `process.nextTick` to schedule reactive work. Async boundaries belong in sources (`fromPromise`, `fromAsyncIter`) and the runner layer. | §5.10 |
| **Raw async primitives (PY)** | Do not use bare `asyncio.ensure_future`, `asyncio.create_task`, `threading.Timer`, or raw coroutines for reactive work. Async boundaries belong in sources (`from_awaitable`, `from_async_iter`) and the runner layer. | §5.10 |
| **Non-central time** | Do not schedule periodic work with raw `setTimeout` / `setInterval` / `time.sleep` for graph-aligned sampling. Use `monotonicNs()` / `monotonic_ns()` for event ordering, `wallClockNs()` / `wall_clock_ns()` for attribution. | §5.11 |

Deferred — all current callers are single-threaded; needs systematic pattern-layer lock audit.

- **`withStatus` decomposition into `statusOf` + `errorOf` (post-1.0, evaluated + deferred 2026-04-27):**
  Surfaced during D.2.3b audit (`SESSION-public-face-blocks-review.md` § F.7). Proposal: decompose `withStatus(src, opts?) → { node, status, error }` into two primitive companion-deriving operators + a thin bundle:

3. **Always optional (user-pluggable):** Persistent data structures (option C — RRB-tree for lists, HAMT for maps) via user-supplied backend. Users opting into persistent semantics pay the bundle cost; everyone else gets the same small core they have today.

**Cross-cutting semantic finding — subscriber callback delivery contract:**
Verified by stress test `S7`. The `downWithBatch` delivery rules (`batch.ts:245-310`) — not previously surfaced in user-facing docs — are:

1. **Within one `down()` call, same tier** → messages coalesced into one array, delivered in one `sink()` call. E.g., `down([[DIRTY], [DIRTY]])` produces one `sink([[DIRTY], [DIRTY]])`.
2. **Within one `down()` call, different tiers** → one `sink()` per non-empty tier group (immediate, deferred, terminal). E.g., `down([[DIRTY], [DATA, 42]])` produces `sink([[DIRTY]])` then `sink([[DATA, 42]])`.
3. **Across different `down()` calls (same batch)** → always separate `sink()` invocations, never merged. Each queued closure in `pendingPhase2` is its own sink call.
4. **No semantic dedup** of repeated idempotent messages (e.g., 5 DIRTYs stay 5 DIRTYs).

Implication: `pushSnapshot()` today uses two separate `down()` calls (one for DIRTY, one for DATA), producing 2 sink invocations per mutation. Even rewriting to a single `down([[DIRTY], [DATA, v]])` produces 2 sink invocations (tier split). Batching helps with ordering/glitch-freedom and DATA drain timing — it does NOT reduce callback count. Bulk APIs (`appendMany`, potential `setMany`) are the only way to collapse N mutations into 1 DIRTY + 1 DATA pair (2 callbacks total instead of 2N). This contract should be documented in the `subscribe` / `node.down` JSDoc.
