## Active work items

- **switchMap-inner teardown hardening for `refineExecutor` / `evalVerifier` (opened 2026-04-23, QA):**
  Inner `refineLoop` and evaluator subgraphs are constructed inside the switchMap project fns of `refineExecutor` / `evalVerifier` and never registered on the parent `HarnessGraph`. When `harness.destroy()` fires, switchMap's subscription teardown drains observers, but any user-supplied `evaluator` that holds its own pending Promise (e.g. an LLM teacher call) can resolve AFTER destroy and attempt to write to a torn-down slot. A regression test (`destroy() after first convergence does not throw on late inner-loop drain`) landed alongside the v1 to catch catastrophic regressions — a fuller hardening pass should (a) add a `dispose` hook to `HarnessExecutor` / `HarnessVerifier` so factories can wire explicit teardown, (b) have `refineLoop` track its in-flight `switchMap` Promise so late resolutions are suppressed rather than written through, and (c) add a stress test that rapidly supersedes items to exercise the switchMap cancel path under load.
  **Deferred** until a real user hits it — today's ephemeral inner graphs clean up via standard subscription teardown, and the regression test catches the obvious failure mode.

- **Router spread order defaults to intake-wins (done 2026-04-23, QA):**
  `routerInput` effect in `harness/loop.ts` now computes `{ ...classification, ...intakeItem }` so intake fields override any stray classification fields an LLM returns. Kept as a log entry because the previous order (`{ ...intakeItem, ...classification }`) was a latent safety hole: a triage prompt update that started returning `affectsAreas` or `_reingestions` would have silently clobbered the harness's reingestion tracking. Watch for any future triage prompt that LEGITIMATELY needs to override an intake field — the fix is to carve out that field in a `pick()` instead of re-flipping the spread.

- **Harness `executeAndVerify` unified slot — escape hatch for tight execute/verify coupling (opened 2026-04-23, refineExecutor design):**
  The 2-slot `executor` / `verifier` interface preserves the ergonomic `executePrompt` / `verifyPrompt` sugar and lets users swap just one side (e.g. refineLoop executor + LLM verifier). But it forces redundant work when execution already produces verification evidence: e.g., an executor that internally runs an eval still has to re-run it in the verifier, because execute output is only `ExecuteOutput = {outcome, detail}`, not the raw eval scores.
  **Shape:** add `executeAndVerify?: (input: Node<TriagedItem | null>) => Node<VerifyResult | null>` option; when set, ignore `executor` + `verifier` and replace the whole `executeNode + executeContextNode + verifyNode` subgraph. Eliminates the rule-4 diamond footgun (no `withLatestFrom` pairing). Semantically the user's pipeline produces full `VerifyResult`s directly.
  **Why deferred:** no concrete consumer yet. The hello example uses 2-slot; the flagship dogfood example uses 2-slot. Add when a real user asks for it or when we catch a redundant-eval pattern in the wild.
  **Design note:** when added, the option name must make the tradeoff obvious — "use this only if your execute + verify are one reactive pipeline; prefer `executor` + `verifier` otherwise." JSDoc + a code comment near the harness slot selection should point users back to the 2-slot default.

- **Harness executor/verifier dev-mode sanity check — assert ≤1 DATA per input wave (opened 2026-04-23, refineExecutor design):**
  The `executor` option's contract (rules 1–4 in the design note) is typed as `Node<ExecuteOutput>` — the type system can't enforce "emits at most once per input item" or "fires on result, not on input." Users who write custom executors can silently violate rule (4) "fire on result, not on input" and burn verification budget without noticing.
  **Shape:** in dev mode (gated by `HarnessLoopOptions.debug ?? process.env.NODE_ENV !== "production"`), wrap the executor output in a monitor that counts DATA emissions per `executeInput` wave and `console.warn` when count > 1. Same for verifier. Near-zero overhead, opt-out, catches misuse during development only.
  **Why deferred:** first-shipped `refineExecutor` makes rule 4 structurally unreachable for its consumers; premature to harden against a misuse pattern we haven't seen. Add once an external user ships a custom executor.

- **`refineLoop` persistent re-seed / reset surface (long-term follow-up to `refineExecutor`, opened 2026-04-23):**
  `refineExecutor` ships in the harness EXECUTE slot with per-item `switchMap` lifecycle (Option A — per-item mount-and-cancel-on-supersede). This is clean but sacrifices cross-item learning: the strategy model (e.g. `errorCritique`'s accumulated failure samples) resets on every triaged item; convergence priors can't carry over from related items with the same `rootCause`. Option B (persistent loop + re-seed) is the long-term best shape.
  **Long-term fix:** add a persistent-session surface to `refineLoop`:
  - `refineLoopGraph.setSeed(next: T)` — swap seed, reset `iteration` to 0, append a session-boundary marker to `history`, keep `strategy` scaffolding intact. Must fire *before* `iterationTrigger` inside one `batch()` so the next generate wave reads the fresh seed — mirrors the `lastFeedback` state-mirror pattern already in `decideEffect` (COMPOSITION-GUIDE §32).
  - `refineLoopGraph.reset(mode?: "soft" | "hard")` — soft = keep strategy object + any effectivenessTracker totals; hard = full state wipe.
  **`refineExecutor` then flips** from `switchMap(input, item => mountFreshRefineLoop(...))` to `effect([input], ({item}) => loop.setSeed(config.seedFrom(item)))` — one persistent loop, sparse re-seed events. Benefits: (a) cross-item learning — strategies like `errorCritique` accumulate failure samples across triaged items sharing a `rootCause`, (b) cheaper — no graph mount/unmount per item, (c) richer `describe()` — one inner loop visible instead of N ephemeral mounts, (d) composes with `autoSolidify` (doc below) more cleanly: patterns learned across items are exactly the promotion candidates.
  **Design work required:** (a) history-per-item vs history-across-items semantics — probably `historyByEpoch: Node<readonly EpochHistory<T>[]>` where one epoch = one seed; (b) `best` semantics across re-seed — reset to null, or persist as all-time best with an `epochBest: Node<T | null>` companion; (c) `attachStorage` checkpoint shape must encode epoch boundaries so restore is deterministic; (d) interaction with `pause()` — does re-seed on a paused loop queue until resume? (e) `setStrategy` already exists and is orthogonal — confirm they compose.
  **Blocked on:** `refineExecutor` (Option A) shipping first so we have a concrete consumer driving the re-seed API. Schedule as a follow-up session once the harness dogfood example surfaces real cross-item learning opportunities worth the design surface cost.

  Landed at [src/patterns/refine-loop.ts](../src/patterns/refine-loop.ts). Topology: 4 `TopicGraph`s (GENERATE / EVALUATE / ANALYZE / DECIDE) over a reactive feedback loop driven by a single `iterationTrigger: state<number>`. `RefineStrategy<T>` interface + `blindVariation` built-in shipped; `errorCritique` / `mutateAndRefine` / registry / `optimizeCatalog` / `refineExecutor` (Composition E) deferred to post-v1. Evaluator shape 4: both `candidates` and `dataset` reactive; dataset arrays auto-wrapped in `state()`. Convergence: patience / min-score / min-delta / max-evaluations / max-iterations / budget. Human-in-the-loop via `setStrategy(next)` + `pause()` / `resume()`.
  **Composition invariants honored:**
  - §5.10 no raw async primitives: the original `switchMap + fromAny(Promise.resolve(…))` bridge was replaced with a producer node that emits synchronously for sync strategies and only crosses a Promise boundary when the strategy itself returns one — eliminates the per-iteration microtask for sync paths and the per-topic iteration-race it would have introduced. Cancellation still works: switchMap unsubscribes the inner producer on new trigger, and the async branch uses a local `cancelled` flag so late Promise resolutions don't emit into a torn slot.
  - §7 feedback cycle: only `iterationTrigger` is a reactive dep of the generate stage; strategy / feedback / prev-candidates / dataset read via closure-seeded updaters (§28), so mid-run swaps apply to the NEXT iteration — never retrigger the current.
  - §28 factory-time seed for `latestStrategy` / `latestFeedback` / `latestPrevCandidates` so the first activation doesn't drop the multi-dep initial pair.
  - §32 state-mirror: the decide-effect writes `lastFeedback.emit(fb)` BEFORE `iterationTrigger.emit(i+1)` inside one `batch()`. The closure updater for `latestFeedback` drains between them, so by the time the next generate's switchMap fires, the closure is fresh. `lastFeedback` is mirrored regardless of decision (continue / converged / budget / paused) so resume-after-pause gets current feedback — previously the pause branch skipped the mirror, leaving stale feedback in the closure.
  - Inline convergence checks inside `decideEffect` rather than reading `convergedNode.cache` — the derived convergence nodes would require a full drain round-trip before their cache updated (they're downstream of `historyState` / `scoreState`, which `decideEffect` itself writes → §7-style write-read deadlock in one wave). The derived nodes still exist for `describe()` / `observe()` surface consistency. **Budget accounting** lives solely in `decideEffect` (single authority — previously split between `generateEffect.emit(budgetState, +cs.length)` and `decideEffect` reading `.cache + 0`, which was sink-order-dependent).
  - Error handling: the candidates producer has a top-level `try/catch` + `actions.down([[ERROR, err]])` surfacing strategy throws as tier-3 ERROR on `candidatesNode`; a dedicated `errorWatcher` effect with `errorWhenDepsError: false` promotes that to `status = "errored"` (previously the `"errored"` status literal existed but was never assignable by any code path — unhandled rejections would leave `status` stuck at `"running"`).
  - De-dup: `lastDecidedIteration` guard so per-iteration decide fires exactly once even if `feedback` / `scores` / `candidates` re-emit within one wave.
  - `pickBest` returns `T | null` on empty candidate batches (COMPOSITION-GUIDE §3 — `undefined` is the protocol SENTINEL; `null` is the domain "no value" signal). `Iteration<T>.best` and `RefineLoopGraph.best` widen to `T | null` accordingly.
  - `isNode` type-guard checks all three core Node methods (`subscribe` / `down` / `emit`) as callable properties, not just `"subscribe" in x` — rejects mock-adjacent objects.
  - `generateEffect` wraps its three emissions (`generateTopic.publish` + `prevCandidatesState.emit`) in a single `batch()` so they coalesce; `resume()` similarly wraps `pauseState` / `statusState` / `iterationTrigger` in one batch and early-returns unless `statusState.cache === "paused"` (idempotent — converged / budget / errored are permanent).

  **Original design shape (kept below for history):**
  Current `fixtures?: string | readonly FallbackFixture[] | StorageTier` stuffs three distinct concerns into one shape-discriminated option. Pre-1.0 refactor — split into three fields, mutually exclusive:
  ```ts
  fixtures?: readonly FallbackFixture[];   // hand-authored inline (messages-keyed OR hash-keyed)
  fixturesDir?: string;                    // directory of cache-format files — auto-namespaced
  fixturesStorage?: StorageTier;           // advanced: custom backend (sqlite/indexeddb/remote)
  ```
  **Why:** `fixtures` unambiguously means "inline". `fixturesDir` is the filesystem happy path with auto-namespacing baked in. `fixturesStorage` is the bring-your-own-backend escape hatch, clearly marked advanced.
  **Messages-keyed stays in `fixtures`** — hash-by-hand is dev-hostile. Hash-keyed lives in `fixturesDir` because that's the record-mode file format (copy-pastable filenames from prior record runs).
  **Directory mode: hash-keyed only** — a hand-authored `{messages, response}` file dropped into `fixturesDir` is silently missed today. Add a ~10-LOC init-time validator: scan the dir, parse a sample, throw a clear `TypeError` if shape doesn't match `CachedEntry`. Document clearly. If a user wants hand-authored in a directory, the fix is: hoist to `fixtures: [...]` and let the adapter compute hashes in memory.
  **Alternatives considered:** (A2) auto-convert messages-keyed files via pre-scan wrapper — breaks "file format matches withReplayCache" promise; write-conflict ambiguity with record mode. (A4) hybrid merged tier — same issues plus more code. (A3) two separate directories — more config surface.

- **`fallbackAdapter` + `withReplayCache` directory commingling — auto-namespace `fixturesDir` (opened + resolved 2026-04-21, inbox-reducer QA):**
  **DONE.** Shipped as B2 in the same session. `fixturesDir: "./dir"` now resolves internally to `fileStorage(join("./dir", keyPrefix))` — multiple adapters sharing a root directory get distinct subdirs per `keyPrefix`. Same auto-namespacing applied to `record.dir`. `fixturesStorage` (B5) preserved as the escape hatch for custom backends (sqlite/indexeddb/cascading) and for opting out of auto-namespace. `record.dir` defaults to `fixturesDir` when omitted — enables the "read baseline, append misses to same dir" pattern. Below kept for history.
  **Original design shape (kept below for history):**
  Two adapters pointed at the same `dir` with different `keyPrefix` (e.g. `fallback:` vs `llm-replay:`) produce distinct keys so lookups don't collide — but files commingle. Bulk-ops (rm, diff, backup scripts) can't tell tenants apart.
  **Recommendation: auto-namespace when `fixturesDir` / `record.dir` is a string → `fileStorage(join(dir, keyPrefix))`.** This becomes the default: physical separation, zero commingle risk.
  **`fixturesStorage` (StorageTier) is NOT just for back-compat** — it's the door for every non-filesystem backend (sqlite / indexeddb / remote / cascading), which a pure `fixturesDir` can't express. Users who want to OPT OUT of auto-namespacing (e.g., interop with an existing fixture set) can reach for `fixturesStorage: fileStorage(dir)` directly. 95% of users never need this; the `fixturesDir` default just works.

  Discovered during the B3 agentLoop rewrite. Repro: a node `T` with deps `[A, B]` where a sink of `B`'s wave (e.g. an `effect`) runs `batch(() => A.emit(...))` inside its callback.   (ii) Amend `_emit` to defer nested batch drains until the parent wave's sink iteration completes. Removes the footgun but may break existing factories that rely on sync drain.

## Implementation anti-patterns

Cross-cutting rules for reactive/async integration (especially `patterns.ai`, LLM adapters, and tool handlers). **Keep this table identical in both repos' `docs/optimizations.md`.**

| Anti-pattern | Do this instead | Spec ref |
|--------------|-----------------|----------|
| **Polling** | Do not busy-loop on `node.get()` or use ad-hoc timers to poll for completion. Wait for protocol delivery: `subscribe` / `firstValueFrom` / `first_value_from` patterns on the node produced by `fromAny` / `from_any`. | §5.8 |
| **Imperative triggers** | Do not use event emitters, callbacks, or `setTimeout` + manual `set()` to trigger graph behavior. All coordination uses reactive `NodeInput` signals and message flow through topology. If you need a trigger, create a reactive source node. | §5.9 |
| **Raw Promises / microtasks (TS)** | Do not use bare `Promise`, `queueMicrotask`, `setTimeout`, or `process.nextTick` to schedule reactive work. Async boundaries belong in sources (`fromPromise`, `fromAsyncIter`) and the runner layer. | §5.10 |
| **Raw async primitives (PY)** | Do not use bare `asyncio.ensure_future`, `asyncio.create_task`, `threading.Timer`, or raw coroutines for reactive work. Async boundaries belong in sources (`from_awaitable`, `from_async_iter`) and the runner layer. | §5.10 |
| **Non-central time** | Do not schedule periodic work with raw `setTimeout` / `setInterval` / `time.sleep` for graph-aligned sampling. Use `fromTimer` / `from_timer` (or other documented `extra` time sources) and compose reactively. Use `monotonicNs()` / `monotonic_ns()` for event ordering, `wallClockNs()` / `wall_clock_ns()` for attribution. | §5.11 |
| **Hardcoded message type checks** | Do not hardcode `if (type === DATA)` for checkpoint or batch gating. Use `messageTier` / `message_tier` utilities for tier classification. | §5.11 |
| **Bypassing `fromAny` / `from_any` for async** | Do not one-off `asyncio.run`, bare `.then` chains, or manual thread sleeps to bridge coroutines / async iterables / Promises into the graph. Route unknown shapes through `fromAny` / `from_any` so `DATA` / `ERROR` / `COMPLETE` stay consistent end-to-end. | §5.10 |
| **Leaking protocol internals in Phase 4+ APIs** | Domain-layer APIs (orchestration, messaging, memory, AI, CQRS) must never expose `DIRTY`, `RESOLVED`, DepRecord, or settlement internals in their primary surface. Use domain language. Protocol access available via `.node()` or `inner`. | §5.12 |

**Cross-cutting semantic finding — subscriber callback delivery contract:**
Verified by stress test `S7`. The `downWithBatch` delivery rules (`batch.ts:245-310`) — not previously surfaced in user-facing docs — are:

1. **Within one `down()` call, same tier** → messages coalesced into one array, delivered in one `sink()` call. E.g., `down([[DIRTY], [DIRTY]])` produces one `sink([[DIRTY], [DIRTY]])`.
2. **Within one `down()` call, different tiers** → one `sink()` per non-empty tier group (immediate, deferred, terminal). E.g., `down([[DIRTY], [DATA, 42]])` produces `sink([[DIRTY]])` then `sink([[DATA, 42]])`.
3. **Across different `down()` calls (same batch)** → always separate `sink()` invocations, never merged. Each queued closure in `pendingPhase2` is its own sink call.
4. **No semantic dedup** of repeated idempotent messages (e.g., 5 DIRTYs stay 5 DIRTYs).

Implication: `pushSnapshot()` today uses two separate `down()` calls (one for DIRTY, one for DATA), producing 2 sink invocations per mutation. Even rewriting to a single `down([[DIRTY], [DATA, v]])` produces 2 sink invocations (tier split). Batching helps with ordering/glitch-freedom and DATA drain timing — it does NOT reduce callback count. Bulk APIs (`appendMany`, potential `setMany`) are the only way to collapse N mutations into 1 DIRTY + 1 DATA pair (2 callbacks total instead of 2N). This contract should be documented in the `subscribe` / `node.down` JSDoc.
