# DS-13.5.C — MemoryRetrievalGraph keepalive + state plumbing

**Status:** LOCKED 2026-05-01 (alternative A: per-call subgraph + keepalive-on-projection-TEARDOWN).

**Scope summary:** Three surgical fixes to `MemoryRetrievalGraph`. **Not a rebuild.** Original DS scope predated the C1 rework — concurrent-isolation and per-call-subgraph mounting are already shipped. Real remaining issues:

1. **Keepalive leak** — per-call subgraph at `retrieve_${id}` never calls `keepalive()`; caller holding `const r = retrieveReactive(q)` without subscribing leaks the subgraph.
2. **Raw `node()` skips equals plumbing** on `context` and `result` (only `projection` has custom `packedEquals`).
3. **`_contextNode` raw + unregistered** when `opts.context` not supplied — invisible to describe.

**No surface API change.** `retrieveReactive(queryInput)` keeps signature.

## COMPOSITION-GUIDE pointers

- **Rule G.7 (feedback cycles)** — per-call subgraph deps must not feed back into shared parent state.
- **Rule G.20-cleanup-default** — TEARDOWN-on-projection disposer fires cleanup correctly under the new auto-precede-COMPLETE behavior (DS-13.5.A Q16).
- **Rule P.5 (factory wiring order)** — wire keepalive disposer AFTER `this.mount(segment, sub)` so the disposer's `this.remove(segment)` succeeds.
- **Rule P.5-keepalive** — Keepalive subscriptions activate computation chain. The TEARDOWN-on-projection pattern is the inverse: tied to LAST unsubscribe, not first.
- **Rule G.4 (Versioned navigation)** — if `result` carries `Versioned`-shaped values, `equals` plumbing on the `state(...)` cell must be appropriate (likely identity-based since result snapshots already have version-shaped dedup elsewhere).

